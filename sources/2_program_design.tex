\subsection{Discussion}
\begin{itemize}
    \item We have chosen not to include getter and setters in the class diagram, since they simply consist of the field name followed by \texttt{get} or \texttt{set}. This is the same way Hubert does in his videoes. Also there are no list of related objects, because they are represented by the arrows. All these conventions simply the diagram greatly, which is much needed, and the purpose of the class diagram.
    \item Also of course there is no list of references to the related objects, becuase these are represented by the arrows.
    \item There is no references to Date, it's a type we declare.
    \item We have chosen to make activities store all the tickets, because in the project description it was important to quickly access the project report and not necessarily for employees to get quick acces to their tickets. We could maintain a list of the employees own tickets, but this would complicate the design since we would have to update it every time we change the tickets. We have chosen to have slightly slower computation time, if that feature is ever implemented.
    \item The projectManagementsystem.app and projectManagementSystem.domain are simply folders containing the objects. 
    \item We chose to make some of the lines in the class diagram oblique, because we found that square lines could be extremely confusing. This is in contradiction to the UML specification, but we found pictures of other people who did it the same way online.
\end{itemize}
